# Feature 2: GitHub OAuth Authentication, Roles & Access Control

## What This Is
Authentication and authorization layer for the platform. Users sign in exclusively via GitHub OAuth. The API issues JWT tokens that are used by both the frontend app and Claude Desktop (via MCP) to call the Feature 1 proxy API.

## Architecture
```
User → Your App → GitHub OAuth → Your API → issues JWT
                                                ↓
Claude Desktop MCP config gets the JWT
                                                ↓
Claude → Your API (Feature 1) → validates JWT → Supabase
```

No passwords. No email/password registration. GitHub is the sole identity provider.

## Tech Stack
- .NET 8 Web API
- Npgsql + Dapper for Postgres access
- JWT Bearer authentication
- GitHub OAuth2 for user identity
- Supabase Postgres (platform schema)

    is_active BOOLEAN DEFAULT true,
    created_at TIMESTAMPTZ DEFAULT now(),
    updated_at TIMESTAMPTZ DEFAULT now()

    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    name TEXT NOT NULL,
    permissions TEXT[] NOT NULL,

    created_at TIMESTAMPTZ DEFAULT now(),

);

-- ─── PROJECT MEMBERS ───────────────────────────────────

    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

## API ENDPOINTS

### Auth (GitHub OAuth)
- `GET /api/auth/github` — Redirect user to GitHub OAuth consent screen
  → Generates random `state` param, stores in session/cookie, redirects to:
  `https://github.com/login/oauth/authorize?client_id={id}&redirect_uri={uri}&scope=user:email,repo&state={state}`

- `GET /api/auth/github/callback?code={code}&state={state}` — GitHub callback
  → Validates state, exchanges code for GitHub access token, fetches GitHub user profile, creates or updates user in `platform.users`, assigns "user" role if new, issues JWT + refresh token, redirects to frontend with tokens

- `POST /api/auth/refresh`
  ```json
  { "refreshToken": "..." }
  ```
  → Issues new JWT + rotates refresh token

- `POST /api/auth/logout` — Revoke refresh token

- `GET /api/auth/me` — Get current user profile + system roles + projects

### Auth Flow Detail
1. Frontend opens `GET /api/auth/github`
2. API redirects to GitHub consent screen
3. User authorizes → GitHub redirects to callback with `code`
4. API exchanges `code` for GitHub access token via `POST https://github.com/login/oauth/access_token`
5. API calls `GET https://api.github.com/user` with token to get profile (id, login, name, email, avatar_url)
6. Lookup `platform.users` by `github_id`:
   - Exists → update `github_username`, `avatar_url`, `github_access_token`, `updated_at`
   - New → insert user, assign "user" role
7. Issue JWT (containing user ID, projects, permissions) + refresh token
8. Redirect to frontend: `https://your-app.com/auth/callback#access_token={jwt}&refresh_token={rt}`

### GitHub OAuth Scopes Requested
- `user:email` — read user's email
- `repo` — read/write repos (needed for Claude to create repos and push code)

### Token Generation for Claude Desktop
- `POST /api/auth/claude-token` — Generate a long-lived token scoped to a specific project
  ```json
  { "projectId": "proj-uuid" }
  ```
  → Returns a JWT with that single project's schema + permissions. This is what the user puts in their Claude Desktop MCP config. Longer expiry (e.g. 30 days) with option to revoke.

  Response:
  ```json
  {
    "success": true,
    "data": {
      "token": "eyJhbG...",
      "expiresAt": "2025-05-01T00:00:00Z",
      "mcpConfig": {
        "mcpServers": {
          "your-platform": {
            "command": "npx",
            "args": ["-y", "your-mcp-package"],
            "env": {
              "API_URL": "https://your-api.com",
              "API_TOKEN": "eyJhbG..."
            }
          }
        }
      }
    }
  }
  ```
  → The `mcpConfig` is a ready-to-use snippet the user can paste into their `claude_desktop_config.json`

- `GET /api/auth/claude-tokens` — List active Claude tokens for current user
- `DELETE /api/auth/claude-tokens/{tokenId}` — Revoke a Claude token

### Role Management (platform_admin only)
- `GET /api/roles` — List all system roles
- `POST /api/roles/assign`
  ```json
  { "userId": "...", "roleName": "platform_admin" }
  ```
- `DELETE /api/roles/revoke` — Remove system role from user

### Projects
- `POST /api/projects` — Create project
  ```json
  {
    "name": "My SaaS App",
    "description": "Customer management platform"
  }
  ```
  → Creates project, provisions Postgres schema, creates default project roles, assigns creator as owner

- `GET /api/projects` — List user's projects
- `GET /api/projects/{id}` — Get project details + members
- `PUT /api/projects/{id}` — Update project (owner only)
- `DELETE /api/projects/{id}` — Deactivate project (owner only)

### Project Members
- `POST /api/projects/{id}/invite`
  ```json
  { "githubUsername": "dev-user", "role": "developer" }
  ```
  → Looks up user by GitHub username, adds as project member

- `PUT /api/projects/{id}/members/{userId}/role`
  ```json
  { "role": "viewer" }
  ```
- `DELETE /api/projects/{id}/members/{userId}` — Remove member
- `GET /api/projects/{id}/members` — List members + roles

### Project Roles (project owner only)
- `GET /api/projects/{id}/roles` — List project roles
- `POST /api/projects/{id}/roles` — Create custom role
  ```json
  { "name": "qa_tester", "permissions": ["read"] }
  ```
- `PUT /api/projects/{id}/roles/{roleId}` — Update permissions
- `DELETE /api/projects/{id}/roles/{roleId}` — Delete custom role

## PROJECT CREATION FLOW

When `POST /api/projects` is called:
1. Generate short ID (e.g. `abc123`)
2. Create record in `platform.projects` with `schema_name = "project_abc123"`
3. Run `CREATE SCHEMA IF NOT EXISTS "project_abc123"` in Postgres
4. Create default project roles in `platform.project_roles`:
   - owner: `["read", "write", "ddl", "manage_members", "delete_project"]`
   - developer: `["read", "write", "ddl"]`
   - viewer: `["read"]`
5. Add creator as member with "owner" role
6. Log action in `platform.audit_log`
7. Return project details + updated JWT

## SECURITY REQUIREMENTS
1. **GitHub OAuth state** — Random `state` param to prevent CSRF
2. **GitHub token encryption** — Encrypt `github_access_token` at rest in the database using AES-256
3. **Token expiry** — App JWT: 60 min. Refresh: 7 days. Claude token: 30 days.
4. **Refresh token rotation** — Old token revoked when new one issued
5. **Claude token revocation** — Users can revoke Claude tokens at any time
6. **Permission checks** — Verify project membership + permission on every operation
7. **Owner-only actions** — Project deletion, member management gated to owner role
8. **Admin-only actions** — System role assignment gated to platform_admin
9. **Audit logging** — Log all auth events, role changes, invitations, project operations
10. **Rate limiting** — Per user, per endpoint
11. **No Supabase credentials exposed** — Only the API server knows the connection string
